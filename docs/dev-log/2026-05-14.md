The big yak-shave of the day. Final state: `public/seal.png` (488×488 ink in a ~680px canvas, white background, two concentric rings at 4px each), rendered at 65×65pt in the signature row next to "For MOLVEXA / Managing Partner".

Journey through the bug:

1. **Initial extract** — cropped the round seal out of `~/Downloads/WhatsApp Image 2026-05-09 at 11.27.05.jpeg` using PIL bbox detection.
2. **Top-arc clipping** — first bbox cut at the source image edge; widened crop and threshold to capture the faint outer ring fully.
3. **Stray "F"** — boundary at x=683 caught the left edge of "For MOLVEXA" text. Bound to x=660.
4. **Wrong intuition: transparency** — tried RGBA with alpha ramp; user wanted white background like the original. Reverted.
5. **"Still cropped"** — turned out the dev server had cached the OLD seal in the Node process. PDF was fine when re-fetched from prod; user was looking at stale localhost output until they restarted `pnpm dev`.

PDF viewers alpha-blend that and the line literally disappears at certain zooms. Thickened to 10px, then dialed back to 8px, 6px, finally 4px per user request to match the inner ring.
7. **Equalized rings** — both outer and inner concentric circles now drawn at 4px so they read as the same weight. Outer ring was also pushed outward by 8px to widen the visible gap with the curved "MOLVEXA / NEAR ARAMKOTTAM L P SCHOOL" text band.

## Key debugging lessons (so we don't relearn them)

### react-pdf caches images by path within the Node process

Changing a file in `public/` won't update PDFs from a long-running `pnpm dev` session — the Node process holds the old buffer. **Restart the dev server after touching any image asset.** Prod deploys are fine because each cold start re-reads.

### PDF sub-pixel rendering kills thin strokes

A line that's less than 1 screen pixel wide at 100% zoom will alpha-blend to a faint gray that PDF viewers will visibly drop at certain zoom levels. Rule of thumb: source image stroke width × (display_pt / source_width) × 96/72 must be ≥ 1. For our 65pt seal in a ~680px source, that means stroke ≥ ~8 source pixels for guaranteed visibility on a standard 96dpi display. Retina/HiDPI is more forgiving (4px works there).

### `pdftoppm -r 200` hides sub-pixel issues

When verifying PDFs locally, render at 200dpi or higher and you'll see thin strokes that the user's PDF viewer at 96dpi cannot. Always sanity-check at 96dpi if a user reports "drop out" or "cropping" of thin lines.

### File size is a useful canary

Two PDFs of the same quote producing different byte counts means different content embedded. When the user's `Q-2026-009.pdf` was 180834 bytes and a fresh prod pull was 197037 bytes, that was the definitive proof the user's local file was stale, not a server-side bug.
